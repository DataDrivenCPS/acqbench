Key findings

  • Per-app overhead is the runtime, ~110–120 MB. An idle read-only app still loads all of acquirium (pyarrow, polars, duckdb), so memory — not CPU or the task — sets the ceiling.
  • Both are memory-bound at ~15 GB under the 16 GB cap: main fits 150 apps, ums-ray 125. Roughly the same wall, reached different ways.
  • main packs more via shared image pages — 150 identical containers share read-only layers, so effective per-app drops toward 100 MB at the top.
  • ums-ray is lighter per app and starts faster (65 vs 75s at 100), but its actors do not share as much, so it caps sooner.
  • The failures differ: at 175, main's nested Docker daemon cannot start a container within its 60s timeout (surfaced as HTTP 400); ums-ray simply cannot bring all 150 online within budget (146 up). Neither is an acquirium bug — both are the infrastructure hitting the memory wall.
