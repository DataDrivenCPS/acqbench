In-App Task vs Concurrency

As the fleet grows to 125 apps, each app's own compute does not slow down at all. What cracks is the shared server read — and only near the memory ceiling, sharply on main, gently on ums-ray.

Key findings

  • Per-app compute is isolated. A ~2.5 ms task stays ~2.5 ms from 1 to 125 apps on both branches. 125 apps ticking once a second demand only ~0.3 of the 4 cores, so simple apps never touch the CPU wall.
  • The shared server read is the real limit. It holds ~17 ms to 100 apps, then main cliffs at 125 (254 ms median, 463 ms p95) as it nears its 150-app memory ceiling; its tail already frays at 100 (p95 143 ms).
  • ums-ray stays graceful — read is still 26 ms at 125 apps, ~10x better than main at the same count. It degrades on startup (bringing apps online), not on the read path.
  • Bottom line: scaling simple apps is limited by the data-access path and memory, not per-app CPU. Nothing an app does to itself slows down; what slows is talking to the one server they share.
